# Personal AI Learning OS

## 项目介绍

Personal AI Learning OS 是一个面向个人学习与知识整理的本地优先工作区。它以 Conversation 为入口，将原始对话或文本整理为 Proposal，并把经过人工确认的内容沉淀为可持续维护的 Knowledge。

当前项目使用 Next.js、React、TypeScript 与 LocalStorage 构建，重点验证完整的产品工作流、清晰的领域边界和可靠的本地数据体验。

## 为什么开发

日常使用 ChatGPT、Claude、DeepSeek 等工具时，有价值的信息往往散落在不同对话和文本文件中，难以持续整理与复用。本项目希望提供一个简单、可控的个人工作区，让用户保留原始来源、人工审核整理结果，并逐步形成属于自己的知识库。

项目坚持从真实工作流出发：先做好数据导入、人工整理、知识管理和本地持久化，再根据实际需求决定是否引入更复杂的基础设施或智能能力。

## 当前完成

- **Sprint1**：建立从原始文本导入、Demo Analyzer 分析、Proposal 审核到 KnowledgeCard 生成的最小可用闭环。
- **Sprint2**：将 Conversation 升级为可创建、编辑、复制、删除和持续保存的个人工作区。
- **Sprint3**：将单一 KnowledgeCard 升级为支持列表、搜索、排序、分页、详情、编辑、归档和删除的本地知识库。

## Roadmap

- **Sprint4**：完善全局搜索与跨 Conversation、Proposal、Knowledge 的统一发现体验。
- **Sprint5**：增加标签、分类和关联能力，让知识可以被组织成主题网络。
- **Sprint6**：增加本地数据导入、导出与备份恢复，保障个人数据的可迁移性。
- **Sprint7**：完善统计、活动记录与学习回顾，帮助用户理解知识积累过程。
- **Sprint8**：在核心工作流稳定后评估可选 AI 能力，并始终保留人工确认与可追溯来源。
- **Sprint9**：完成生产化准备，包括测试、可访问性、性能、数据迁移和部署方案。

## Development Rules

- 保持 Domain 分层：实体、契约与领域服务放在 `src/core`，不依赖页面或浏览器基础设施。
- 保持 Storage 分层：页面和组件通过 Storage 封装访问数据，禁止直接调用 LocalStorage。
- LocalStorage 优先：在本地数据规模和使用场景足够时，不引入额外持久化设施。
- 不要提前数据库：只有在明确出现多设备同步、协作或数据规模需求后再评估数据库。
- 不要提前 AI：先保证人工工作流完整、可靠、可解释，再引入可选的智能能力。
- 不要过度设计：优先交付当前 Sprint 的最小完整能力，避免为尚未出现的需求建立抽象。
- 保持来源可追溯：Proposal 和 Knowledge 应尽可能保留其原始 Conversation 或文件来源。
- 保持人工控制：整理、接受、归档和删除等关键操作必须由用户明确触发。
- 保持向后兼容：Storage 数据结构调整时应考虑已有 LocalStorage 数据的读取与迁移。
- 保持质量门槛：每个 Sprint 交付前必须通过 lint、类型检查和 production build。
- 保持变更聚焦：一个任务只修改必要范围，不顺带重构无关业务代码。
- 不提交敏感信息：源码、日志和示例数据中不得包含密钥、令牌或个人隐私数据。
