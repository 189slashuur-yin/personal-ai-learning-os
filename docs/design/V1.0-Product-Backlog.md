# v1.0 Product Backlog

## 文档状态

本文是 v1.0 planning 输入，不是已批准需求、Sprint 承诺或实现授权。v0.9 人工 QA 与缺陷分级完成后，产品负责人需要重新确认 v1.0 范围、验收标准和交付顺序。

## v1.0 产品目标建议

v1.0 应把 PALOS 从“功能完整的单浏览器 local-first MVP”推进到“用户敢于持续积累、能够可靠找回与迁移个人资料的稳定本地产品”。重点应是可信数据、精确定位、可恢复性与高频知识复用，而不是扩大集成面。

## v1.0 不做什么

- 不做 Agent、自动执行、自动接受 Proposal 或自动创建/完成/删除 Task。
- 不做 RAG、Embedding、向量数据库、语义问答或知识图谱。
- 不做 Calendar、Reminder、Recurring Task、Pomodoro、Habit 或 Workflow 平台。
- 不做账号、团队、多租户、权限、云同步或多人协作。
- 不在 Web 浏览器中扫描任意文件系统、静默复制文件或承诺管理外部文件生命周期。
- 不以 v1.0 为由进行框架迁移、完整 DDD 化、状态管理替换或大规模目录重写。
- 不直接启用 OpenAI、Claude、Gemini、DeepSeek、Azure 或 Custom 云 Provider，不收集 API Key。
- 不把 LocalStorage 直接替换为数据库；数据库迁移必须建立在导出格式、容量测量和 Adapter 方案之上。
- 不把“项目文档备份脚本”描述为完整用户数据备份。

## v1.0 必须解决什么

| 优先级 | 必须解决的问题 | v1.0 最小完成标准 |
| --- | --- | --- |
| P0 | 用户结构化数据无法完整导出和恢复 | 版本化本地 Export / Import；校验、预览、冲突策略、失败回滚与恢复演练可验证 |
| P0 | v0.9 尚无完成的 release QA 证据 | 50 条 v0.9 人工计划执行完毕；阻塞缺陷清零；关键主链路形成自动化回归基线 |
| P0 | Conversation 与 Asset owner 生命周期未闭合 | 明确定义并验证 copy/delete 行为；不留 orphan metadata，也不隐式删除外部文件 |
| P1 | Search 找到 Message/Q&A 后不能精确定位 | 结果具有稳定 anchor；打开后滚动并高亮目标；失效引用有明确降级 |
| P1 | Search 排序质量与数据规模没有测量 | 固定相关性语料、误命中/漏命中清单、1k/10k 文档性能基准与可接受阈值 |
| P1 | LocalStorage 风险对用户仍不可操作 | Data Management 显示容量/边界，导出前后一致性可核对，失败不清空已有数据 |
| P1 | 知识积累后缺少高频复用闭环 | 选择一个小而完整的 Knowledge Productivity 场景并证明它提高找回或复习效率 |

## 建议优先级

1. Local Data Export / Import
2. Search Precision & Anchors
3. v0.9 QA 自动化基线与稳定性修复
4. Knowledge Productivity 的一个受控最小切片
5. Asset File Handling 只做设计验证或可选本地适配器 spike
6. Real AI Provider UX 只做风险与交互设计，不进入 v1.0 runtime

## Epic 1 — Search Precision & Anchors

### 候选范围

- 为 Message、Q&A Pair、Source fragment、Proposal evidence 和 Knowledge source 增加稳定的结果 anchor。
- 从 Search 打开 owning page 后滚动、聚焦并短暂高亮具体文本单元。
- 建立可重复的中英文相关性语料和 exact / contains / fuzzy 排序断言。
- 记录索引规模、构建时间、查询延迟、结果数量与弱匹配原因。
- 仅在测量证明需要时，评估权重调整、MiniSearch/Fuse.js 或未来桌面 SQLite FTS；不预先引入依赖。

### 用户价值

用户不仅能知道“某个 Conversation 里有结果”，还能直接抵达证据位置；这会显著降低长对话中的找回成本，并增强 Source-first 的可信感。

### 风险

- Q&A Pair 是派生模型，Message 编辑或 Snapshot Restore 后 anchor 可能失效。
- URL 状态、滚动时机、虚拟/折叠视图和高亮可能耦合 Page 实现。
- 为提高召回率而扩大 fuzzy 会损害精确度和用户信任。
- 引入第三方索引库可能增加 bundle、初始化成本和兼容路径。

### 依赖

- v0.9 SearchDocument 的稳定 source identity 与 source path。
- Conversation Timeline/Q&A Pair 可接受 anchor 参数并提供失效降级。
- 固定 QA fixtures、相关性标准和性能数据集。
- 任何索引库选择都需先有量化阈值和迁移/备份说明。

### 是否推荐进入 v1.0

**推荐，P1。** 先做 anchor 与质量基准，再决定是否替换算法；不将语义搜索纳入本 Epic。

## Epic 2 — Asset File Handling

### 候选范围

- 明确区分 external reference 与 managed local asset library。
- 设计用户显式选择文件、复制到受控目录、冲突命名、hash 校验、missing/changed 状态和重新关联。
- 通过独立 CLI 或 desktop adapter 验证路径完整性；Web 页面继续只依赖 Asset Contract/Service。
- 定义 owner 删除、metadata 删除、managed file 删除和备份/恢复的不同确认语义。
- 不在纯浏览器版本中扫描目录或后台读取任意路径。

### 用户价值

用户能知道一个附件是否仍存在、是否被修改、是否包含在备份中，而不是只保存一个可能失效的字符串路径。

### 风险

- 浏览器安全模型无法可靠访问用户输入的任意路径。
- 文件复制和删除是高风险破坏性操作，可能造成重复、丢失或隐私泄露。
- Windows/macOS/Linux 路径、权限、Unicode、移动硬盘和 iCloud 同步语义不同。
- managed files 会把备份体积、hash 计算、冲突处理和恢复引入产品责任。

### 依赖

- ADR-003 的 Asset identity、owner 与 metadata 边界保持稳定。
- v0.9 blocker 先闭合 Conversation copy/delete 与 Asset metadata 生命周期。
- 明确选择 Web、CLI 或 desktop runtime；不能用页面假装拥有文件系统能力。
- Local Data Export / Import 格式需要包含 Asset metadata 与 portable reference。
- 破坏性操作确认、备份 manifest、完整性检查与恢复演练。

### 是否推荐进入 v1.0

**不推荐完整进入 v1.0。** 可做 P2 设计与受控 CLI/desktop spike；v1.0 runtime 最多提供显式 integrity check，不承诺完整文件托管。

## Epic 3 — Real AI Provider UX

### 候选范围

- 设计 Provider onboarding、模型选择、连接测试、费用/隐私提示、错误恢复、速率限制与用量可见性。
- 定义 API Key 的输入、存储、清除和泄露响应；默认不得进入 LocalStorage 或日志。
- 保持 AnalyzerProvider → Proposal → Human Review，不允许模型直接创建 Knowledge 或改变业务状态。
- 先以不可执行 prototype 和 threat model 验证 UX，不默认连接任何云端。

### 用户价值

用户可以选择更强模型分析自己的资料，并理解数据会发送到哪里、可能产生何种费用、失败后如何恢复。

### 风险

- 密钥托管、浏览器暴露、隐私合规、数据出境、费用失控和供应商条款均未解决。
- 不同 Provider 的 schema、限流、模型退役和错误语义会扩大维护面。
- 真实模型输出非确定，可能降低现有 QA 可重复性。
- 一旦 UX 模糊，用户可能误以为原始私密资料始终只在本地。

### 依赖

- 单独批准的安全、隐私、费用、错误处理和验收方案。
- 密钥不落前端持久化的可信运行时或代理设计。
- Provider contract/versioning、审计日志脱敏、测试 doubles 与预算保护。
- 明确哪些数据发送、何时发送，以及用户逐次/持续同意模型。

### 是否推荐进入 v1.0

**不推荐进入 v1.0 runtime。** 仅推荐做 P2 UX prototype、threat model 与架构决策；继续以 Demo 为默认、Ollama 为用户显式启用的唯一真实模型调用。

## Epic 4 — Knowledge Productivity

### 候选范围

- 从“找得到”推进到一个高频复用动作，例如 Focus/Review Queue、复习状态或手动集合；三者只选一个最小闭环。
- 复用现有 Knowledge status、Tag、Workspace、Source provenance 与 Task 显式创建能力。
- 提供清晰的“为什么出现在这里”和“完成后发生什么”，不引入后台催办或 AI 自动行动。
- 用可观测的用户结果验收，例如从搜索命中到复习完成、从 Knowledge 到 Task 的转化。

### 用户价值

用户积累的 Knowledge 不再只停留在归档和搜索，而能进入轻量、可控、可解释的复习或再利用流程。

### 风险

- “生产力”容易膨胀为 Reminder、Calendar、Activity、Agent 或复杂 Task 系统。
- 新状态若直接塞进 KnowledgeCard，会混淆知识内容和行动状态。
- 多种组织方式同时上线会增加认知负担和迁移成本。
- 缺少真实使用证据时可能优化错误工作流。

### 依赖

- 产品负责人选择一个用户问题和一个主要成功指标。
- 先确认是否需要新独立 Entity；不得默认扩充 KnowledgeCard。
- Search anchor 与可靠 Export / Import 支撑长期使用。
- 完整生命周期、删除/归档、旧数据默认值和 QA 方案。

### 是否推荐进入 v1.0

**有条件推荐，P1/P2。** 只批准一个小而完整的场景；若范围扩张到 Reminder、Activity 或 Agent，则移出 v1.0。

## Epic 5 — Local Data Export / Import

### 候选范围

- 通过 Storage Contracts 导出全部浏览器结构化集合，包含 schema/version、导出时间、应用版本、记录计数与校验信息。
- Import 前完成解析、schema 校验、兼容迁移、计数预览和冲突策略选择。
- 支持 dry-run/preview；写入失败不得清空现有数据，必要时先生成回滚快照。
- 明确 replace、merge 或新 profile 的语义；v1.0 可只批准一种最安全策略。
- 导出 Asset metadata，但不谎称包含 external files；可生成 missing external assets 清单。
- 建立 round-trip、旧版本 fixture、损坏文件、超大文件和中断恢复测试。

### 用户价值

用户可以迁移浏览器、保存个人数据副本并验证恢复结果。这是从实验性 LocalStorage MVP 迈向可信长期使用的最直接能力。

### 风险

- 跨集合引用、Conversation 级联关系和 ID 冲突可能导致部分导入或悬空引用。
- Export 文件包含私人对话、知识和本地路径，必须明确敏感性并避免日志泄露。
- merge 语义复杂，错误恢复可能比导出本身更危险。
- schema version 设计若过早耦合当前存储形状，会阻碍未来数据库迁移。

### 依赖

- 盘点全部 Storage Contract、LocalStorage key、旧字段归一化和引用不变量。
- 定义稳定、可验证、与 BrowserStorage 内部 key 解耦的 exchange schema。
- 导入事务/分阶段写入与 rollback 设计。
- 加密不应被暗示为已有；若要加密必须独立设计密钥与恢复策略。

### 是否推荐进入 v1.0

**强烈推荐，P0。** 这是 v1.0 最应优先交付的 Epic；先做可靠 replace/restore 闭环，再考虑 merge、加密或云目标。

## 推荐的 v1.0 切片

若只批准一个可控版本，建议包含：

1. Local Data Export / Import 的版本化 replace/restore 闭环。
2. Message/Q&A Search Anchors 与固定相关性/性能基准。
3. v0.9 release-blocking QA 缺陷修复和关键 E2E/Integration 基线。
4. 一个经过确认的 Knowledge Productivity 最小场景。

Asset File Handling 与 Real AI Provider UX 保持设计/验证状态，不进入默认 runtime。这个组合优先解决“数据会不会丢、内容能不能准确找回、知识能不能再次使用”，同时控制安全和架构扩张。

## 进入实现前的批准门槛

- 每个 Epic 有明确非目标、数据生命周期、兼容策略和删除语义。
- 每个新增字段都有旧数据默认值或迁移方案。
- Export / Import 有隐私提示、失败回滚和 round-trip 验收。
- Provider 若超出 Demo/Ollama，必须先有密钥、安全、隐私、费用和错误处理批准。
- 任何数据库或索引库选择都有实测数据，不以“未来可能需要”为理由提前引入。
- Sprint 开始前把候选范围转换为可验证 acceptance criteria，并更新 ROADMAP；本 Backlog 本身不构成授权。
